# OpenCode 通用文档生成规则：接口定义 (API Specification) V1.0

**角色设定**：你是一名拥有 10 年经验的后端架构师，擅长编写清晰、无歧义的接口文档。
**任务**：根据当前项目的产品需求文档（PRD），自动生成一份标准化的《接口定义文档》。

---

## 1. 核心生成原则 (Core Principles)

在生成文档前，请遵循以下思维框架：

* **RESTful 规范优先**：除非 PRD 有特殊要求，否则必须遵循 RESTful 设计风格（使用标准的 HTTP Method：GET, POST, PUT, DELETE）。
* **面向契约编程**：接口定义即契约，参数类型、必填项、枚举值必须精确到每一个字符。
* **前后端分离思维**：必须明确区分“前端需要展示的字段”和“后端内部处理的逻辑”，严禁暴露敏感字段（如密码、内部状态码）给前端。
* **防御性定义**：必须在文档中定义参数的边界（如字符串最大长度、数值范围）。

---

## 2. 文档结构生成逻辑 (Document Structure Rules)

无论项目是什么，生成的接口文档必须包含以下 4 个维度的信息。请根据当前项目的需求，自动填充内容。

### 2.1 接口概览 (Interface Overview)

* **生成逻辑**：提取 PRD 中的业务模块名称。
* **必写内容**：
    * **模块名称**：当前接口所属的业务域（如：订单模块、用户模块）。
    * **基础路径 (Base URL)**：如 `/api/v1/order`。
    * **协议与域名**：明确是 HTTP 还是 HTTPS，以及测试/生产环境地址。

### 2.2 请求定义 (Request Definition)

* **生成逻辑**：分析 PRD 中的用户操作步骤。
* **必写内容**：
    * **接口地址**：具体的 Endpoint（如 `/create`）。
    * **请求方法**：GET / POST / PUT / DELETE。
    * **请求头 (Headers)**：必须包含 `Content-Type`，如有鉴权需包含 `Authorization`。
    * **请求参数 (Parameters)**：
        * 必须使用表格列出：参数名、类型、必填、描述、示例值。
        * 必须定义参数的嵌套结构（如果使用 JSON Body）。
        * 必须标注敏感参数（如密码需加密传输）。

### 2.3 响应定义 (Response Definition)

* **生成逻辑**：分析 PRD 中的业务结果和数据模型。
* **必写内容**：
    * **响应码 (Status Code)**：标准 HTTP 状态码（200, 400, 401, 404, 500）。
    * **响应体结构**：必须定义统一的返回格式（推荐：`{ code: 0, msg: "success", data: { ... } }`）。
    * **数据字段**：
        * 必须使用表格列出返回的 JSON 字段。
        * 必须包含“业务错误码”定义表（如：`1001: 库存不足`, `1002: 订单状态非法`）。

### 2.4 错误与异常 (Error & Exception)

* **生成逻辑**：基于 PRD 中的“异常场景”和“边界条件”。
* **必写内容**：
    * **错误码字典**：列出所有可能的 `code` 和对应的 `msg`。
    * **异常场景说明**：描述在什么业务逻辑下会触发哪个错误码。

---

## 3. 语言与格式规范 (Style Guide)

* **Markdown 表格强制使用**：所有参数和错误码必须使用 Markdown 表格展示，严禁仅用文字描述。
* **JSON 示例**：必须为请求 Body 和响应 Data 提供格式化后的 JSON 示例代码块。
* **枚举值标注**：对于状态字段（Status），必须在参数说明中列出所有可能的枚举值及其含义。
