#  项目文档生成与审核总纲 (Master Document Guideline)

本文件是项目文档体系的最高指导原则。所有由 AI 生成或人工编写的技术文档（包括但不限于 DDD 设计、系统架构、数据库设计、接口定义、核心算法等），都必须遵循本总纲进行自我审查和正确性校验。

---

## 1. 核心原则

### 1.1 单一事实来源
> **文档是代码的“设计蓝图”，代码是文档的“最终实现”。**

- **一致性强制**：任何设计变更必须同步更新文档，任何代码修改必须符合文档规范。
- **禁止冲突**：严禁出现文档与代码逻辑不一致的情况（如文档说是 `String`，代码里是 `Integer`）。

### 1.2 全链路一致性
所有文档必须形成一个闭环的逻辑链条，严禁出现断层：
1. **需求文档**决定了业务边界。
2. **DDD 文档**决定了业务模型（聚合根、实体）。
3. **架构文档**决定了技术实现路径（微服务、中间件）。
4. **数据库与接口文档**决定了物理存储与交互契约。
5. **核心逻辑文档**决定了具体算法。

> **审查标准**：如果数据库字段不支持接口参数，或者接口定义违背了 DDD 聚合根的定义，则该文档体系视为“不合格”。

### 1.3 防御性设计
所有文档必须预设“失败”场景，不能只写“快乐路径”：
- **接口设计**：必须包含错误码定义和异常处理。
- **数据库设计**：必须包含逻辑删除和审计字段。
- **核心算法**：必须包含边界条件检查和异常处理流程。
- **架构设计**：必须包含容灾、降级和熔断策略。

### 1.4 标准化与规范化
- **命名规范**：所有文档中的命名（表名、字段名、接口名、类名）必须严格遵循项目约定的命名风格（如蛇形命名法或驼峰命名法）。
- **格式规范**：必须使用 Markdown 编写，图表必须使用 Mermaid 语法，确保在 Git 仓库中可直接渲染预览。

---

## 2. 文档生成与审核清单

在生成或提交任何文档之前，必须通过以下维度的自我审查：

| 维度 | 审查问题 | 通过标准 |
| :--- | :--- | :--- |
| **业务一致性** | 文档是否真实反映了 PRD 的需求？ | 所有核心业务流程都有对应的逻辑设计，无遗漏。 |
| **技术可行性** | 设计的技术方案是否可行？ | 选用的技术栈（如 Redis, MySQL）能支撑该设计，无逻辑悖论。 |
| **数据完整性** | 数据库设计是否满足业务查询需求？ | 关键字段类型正确（如金额用 Decimal），包含必要的索引和软删除字段。 |
| **接口契约性** | 接口定义是否清晰且无歧义？ | 请求/响应结构明确，错误码定义完整，前后端联调无障碍。 |
| **架构规范性** | 是否符合分层架构原则？ | 领域层不依赖基础设施层，模块间无循环依赖。 |

---

## 3. 冲突解决机制

当不同文档之间出现冲突时，遵循以下优先级（由高到低）：

1. **产品需求文档**：业务价值最高，如果技术实现无法满足业务需求，需重新设计技术方案。
2. **DDD 架构设计**：定义了系统的核心模型和边界，接口和数据库设计必须服从领域模型。
3. **系统架构设计**：定义了技术约束，数据库和核心算法设计必须服从架构规范。
4. **详细设计文档（数据库/接口/算法）**：必须服从上层设计。

---

## 4. 维护与更新

- **版本控制**：所有文档必须随代码版本一起进行版本控制。
- **变更同步**：修改代码逻辑时，必须同步检查并更新相关文档。
- **废弃标记**：过期的设计文档必须标记为“已废弃”或删除，防止误导后续开发。

---

## 5. 如何使用本规则

- **对于 AI**：在生成任何具体文档（如数据库设计、接口定义）前，请先读取本文件，确保生成的文档符合上述原则。
- **对于人类**：在 Code Review 阶段，请对照本文件中的“审核清单”检查 AI 生成的文档质量。

---

## 6. 文档生成与自我修正流程 (新增核心流程)

为了确保文档的高质量和一致性，所有技术文档的生成必须遵循**“双阶段生成法”**。AI 严禁直接输出未经审查的初稿，必须在生成后自动执行审核与修正。

### 6.1 第一阶段：初稿生成 (Draft Generation)
AI 扮演**“架构师”**角色，根据 PRD 生成文档初稿。
- **输入**：产品需求文档 (PRD)。
- **动作**：生成 DDD 模型、架构图、数据库 Schema 和接口定义。
- **要求**：侧重于“完整性”，确保所有业务场景都被覆盖。

### 6.2 第二阶段：自我审核与修正 (Self-Audit & Correction) [核心]
AI 必须立即切换身份，扮演**“严苛的技术总监”**，基于本总纲对初稿进行强制审查。
- **动作**：
    1. **全链路比对**：自动检查数据库字段是否支撑接口参数，检查命名是否规范。
    2. **防御性补全**：自动检查是否缺失逻辑删除字段、错误码定义或异常处理流程。
    3. **修正执行**：如果发现任何不符合《总纲》第 2 章“审核清单”的地方，必须**直接修改**文档内容，而不是仅仅列出问题。
- **要求**：只有通过了自我审查（或完成了自我修正）的文档，才被视为“可交付物”。

### 6.3 第三阶段：版本化交付 (Versioned Delivery)
通过审核的文档必须按照以下规范交付：
- **文件命名**：`[模块名]_[文档类型]_v[版本号].md` (例如：`User_Service_Architecture_v1.0.md`)。
- **存放位置**：必须存放在项目根目录下的 `docs/` 文件夹或模块的 `doc/` 子目录中。
- **变更日志**：如果更新文档，必须在文件头部更新《修改日志》，记录修改原因。
