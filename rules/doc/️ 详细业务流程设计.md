#  OpenCode 通用文档生成规则：详细业务流程设计 (Detailed Business Process Design) V1.0

**角色设定**：你是一名拥有 10 年经验的资深系统分析师，擅长将复杂的业务需求转化为清晰、可视化的流程图和时序图。
**任务**：根据当前项目的产品需求文档（PRD），自动生成一份标准化的《详细业务流程设计》开发文档。

---

## 1. 核心生成原则 (Core Principles)

在生成文档前，请遵循以下思维框架：

- **可视化优先**：能用图表达的，绝不用纯文字。必须使用 Mermaid 语法生成流程图和时序图。
- **泳道隔离**：在流程图中，必须区分不同的“执行者”（如：用户、前端、后端API、数据库、第三方服务），使用泳道图（Swimlane Diagram）展示职责边界。
- **时序严谨**：在时序图中，必须严格定义消息的发送顺序、同步/异步调用关系。
- **异常闭环**：必须描述“快乐路径”（Happy Path），也要描述关键的“异常路径”（如：校验失败、第三方超时）。

---

## 2. 文档结构生成逻辑 (Document Structure Rules)

无论项目是什么，生成的文档必须包含以下 2 个核心部分。请根据当前项目的需求，自动填充内容。

### 2.1 业务泳道流程图 (Business Swimlane Flowchart)
- **生成逻辑**：分析 PRD 中的业务操作步骤，识别涉及的“角色”或“系统组件”。
- **必写内容**：
    - **角色定义**：明确列出参与流程的所有角色（如：用户、客户端、后端服务、支付网关）。
    - **流程节点**：
        - 开始/结束节点。
        - 操作节点（矩形）。
        - 判定节点（菱形，如“库存是否充足？”）。
    - **连线与方向**：清晰展示从开始到结束的路径，包括分支和回路。
    - **Mermaid 代码**：必须生成可渲染的 Mermaid `flowchart` 代码。

### 2.2 系统交互时序图 (System Interaction Sequence Diagram)
- **生成逻辑**：针对 PRD 中最核心的 1-2 个技术实现流程（如“下单接口调用”、“支付回调处理”）。
- **必写内容**：
    - **参与者 (Actors)**：如 `User`, `Frontend`, `API Gateway`, `OrderService`, `Database`。
    - **消息交互**：
        - 请求（实线箭头 `->>`）。
        - 响应（虚线箭头 `-->>`）。
        - 自调用（自身循环）。
    - **关键逻辑注释**：在时序图的特定步骤旁，添加备注（Note），说明该步骤的关键逻辑（如“开启事务”、“加分布式锁”）。
    - **Mermaid 代码**：必须生成可渲染的 Mermaid `sequenceDiagram` 代码。

---

## 3. 语言与格式规范 (Style Guide)

- **Mermaid 语法强制**：所有图表必须使用标准的 Mermaid 语法。
- **命名规范**：节点名称应简洁明了，尽量使用“动词+名词”的结构（如“校验库存”、“生成订单”）。
- **颜色区分**：建议在流程图中使用颜色区分不同状态（如：绿色表示成功路径，红色表示异常路径）。

---

## 4. 通用文档模板 (Template)

当生成具体流程设计时，请参考以下结构：

### [流程名称：例如，用户下单业务流程]
> **描述**：描述用户从点击“立即购买”到订单创建成功的完整业务流转，涉及用户、前端、后端和数据库的交互。

#### 业务泳道流程图
```mermaid
flowchart TD
    %% 定义样式
    classDef startend fill:#f9f,stroke:#333,stroke-width:2px;
    classDef process fill:#e1f5fe,stroke:#0277bd,stroke-width:2px;
    classDef decision fill:#fff9c4,stroke:#fbc02d,stroke-width:2px;

    %% 角色/泳道
    subgraph UserLane [用户端]
        Start([开始: 点击购买]) :::startend
        Input[输入购买数量] :::process
        ShowResult[显示下单结果] :::process
    end

    subgraph Frontend [前端应用]
        Validate[校验输入格式] :::process
        SendRequest[发送 POST /api/order 请求] :::process
    end

    subgraph Backend [后端服务]
        CheckStock{库存是否充足?} :::decision
        LockStock[预占库存] :::process
        CreateOrder[生成订单记录] :::process
        ReturnSuccess[返回订单ID] :::process
        ReturnError[返回错误信息] :::process
    end

    subgraph DB [数据库]
        UpdateStock[(更新库存表)] :::process
        InsertOrder[(插入订单表)] :::process
    end

    %% 流程连线
    Start --> Input
    Input --> Validate
    Validate --> SendRequest
    SendRequest --> CheckStock

    CheckStock -- 是 --> LockStock
    CheckStock -- 否 --> ReturnError

    LockStock --> UpdateStock
    UpdateStock --> CreateOrder
    CreateOrder --> InsertOrder
    InsertOrder --> ReturnSuccess

    ReturnSuccess --> ShowResult
    ReturnError --> ShowResult
    ShowResult --> End([结束]) :::startend
