## 1. 文档概述

### 1.1 项目背景
本项目聚焦于电商平台最为核心的交易链路闭环。当前阶段需剥离商品详情、用户评价、物流配送等外围模块，单独抽离并构建一个高内聚的“下单与支付”模拟核心域。目标是清晰定义用户从“点击购买”到“支付成功”过程中的业务规则与数据状态流转，为后续整体电商系统构建提供稳定的底层逻辑支撑。

### 1.2 核心目标
- **流程闭环**：实现用户下单、库存预占、订单生成、模拟支付、结果回调、订单完结的全生命周期管理。
- **数据一致性**：在业务层面建立严格的库存控制规则，防止出现“超卖”现象。
- **异常处理定义**：明确针对库存不足、支付超时等逆向场景的业务处理标准。

### 1.3 阅读对象
- 产品经理、业务运营人员、测试人员、架构师。

## 2. 核心业务流程定义

本章节描述系统在理想状态下的正向业务走向。

### 2.1 正常下单与支付流程
本流程涉及两个关键业务动作：**“提交订单”**与**“发起支付”**。

| 步骤 | 业务节点 | 业务行为描述 | 数据状态预期 |
| :--- | :--- | :--- | :--- |
| **1** | **下单校验** | 用户提交包含商品ID、数量的下单请求。系统需核验当前商品的可售库存是否满足本次购买数量。 | 库存充足 -> 继续；库存不足 -> 终止流程。 |
| **2** | **库存预占** | 为保证后续支付成功的用户能拿到货，系统将对应的商品库存标记为“预占/冻结”状态，此部分库存暂时不可被其他订单购买。 | 可售库存减少，冻结库存增加。 |
| **3** | **订单生成** | 系统生成一笔唯一的订单记录，明确记录购买商品、数量、金额及下单时间。 | **订单状态：待付款**<br>**支付状态：未支付** |
| **4** | **用户支付操作** | 用户针对该笔订单发起支付请求。系统据此生成一条记录本次支付意图的凭证。 | 生成支付记录，**支付状态：处理中** |
| **5** | **模拟支付成功** | 模拟外部支付渠道（如银行）向本系统发送“支付成功”的通知。 | 系统更新对应支付记录的最终结果。 |
| **6** | **订单完结** | 系统确认收到支付成功通知后，变更订单状态，并确认库存扣减的有效性。 | **订单状态：已支付/待发货**<br>**库存：冻结库存转化为实际消耗库存** |

### 2.2 支付超时自动取消流程
系统需具备自动时间管理机制，以处理用户下单后放弃支付的情况。

| 触发条件 | 业务规则 | 业务结果 |
| :--- | :--- | :--- |
| 订单状态为“待付款”，且当前系统时间超过 **“订单创建时间 + 15分钟”**。 | 系统判定该交易超时失效。 | 1. **订单状态**变更为“已取消”。<br>2. **库存回滚**：该订单在步骤2中预占的库存需重新释放回可售库存池，以供其他用户购买。 |

## 3. 详细功能需求说明

### 3.1 功能模块：下单管理

| 需求编号 | 功能名称 | 业务规则详述 |
| :--- | :--- | :--- |
| **FN-01** | **库存校验与锁定** | **场景 A（成功）**：若 `下单数量 ≤ 当前商品可售库存`，则执行预占动作，并允许订单生成。<br>**场景 B（失败）**：若 `下单数量 > 当前商品可售库存`，则立即终止下单操作，并向用户反馈明确的提示：“商品库存不足，请调整购买数量”。<br>**约束**：同一商品在同一时刻的并发购买请求，需保证预占库存操作不会突破总库存上限。 |

### 3.2 功能模块：订单状态管理

| 需求编号 | 功能名称 | 业务规则详述 |
| :--- | :--- | :--- |
| **FN-02** | **订单生命周期定义** | 订单在本系统核心域内仅包含以下三个关键状态：<br>1. **待付款**：订单已创建，等待用户付款。<br>2. **已支付**：已收到支付成功确认，交易达成。<br>3. **已取消**：因超时或系统主动触发导致的交易关闭。 |
| **FN-03** | **支付超时检测** | 系统需具备识别“创建时间距今超过 **15分钟** 且状态为 **待付款**” 订单的能力。<br>对于识别出的订单，需自动触发**订单取消流程**。 |

### 3.3 功能模块：模拟支付与回调

| 需求编号 | 功能名称 | 业务规则详述 |
| :--- | :--- | :--- |
| **FN-04** | **支付凭证生成** | 用户发起支付时，系统需生成一笔支付流水记录，记录金额、关联订单号、发起时间。初始状态标记为“等待回调”。 |
| **FN-05** | **支付结果处理** | **正常回调**：收到模拟的“支付成功”通知后，系统需校验通知中金额是否与订单金额一致。<br>**结果更新**：<br>- 支付流水状态变更为“支付成功”。<br>- 关联订单状态变更为“已支付”。<br>- 该笔订单预占的库存正式确认为“已售/已消耗”。<br>**幂等处理**：对于同一笔订单的重复成功回调，系统仅处理第一次，后续通知不再重复变更订单状态，避免数据异常。 |

## 4. 异常与边界场景处理

| 场景编号 | 场景描述 | 系统预期表现 |
| :--- | :--- | :--- |
| **EX-01** | **下单时库存刚好被抢光** | 参照 **FN-01 场景 B**，明确提示用户库存不足，系统不留存未完成的预订单数据。 |
| **EX-02** | **支付回调晚于订单超时** | 订单若已在第15分钟时自动取消，即使第16分钟收到了支付成功的回调，系统应**拒绝**处理此次支付结果，并记录异常日志（提示：订单已关闭）。此时库存已释放，不可再扣减。 |
| **EX-03** | **支付回调金额不符** | 收到的支付成功金额与订单应付金额不一致。系统应标记该笔支付为“异常”，不改变订单状态，并触发人工核查预警。 |

## 5. 核心数据模型定义（业务视角）

此部分描述业务运营过程中需要关注的关键信息字段，非数据库设计。

### 5.1 订单信息
- **订单编号**：唯一识别码。
- **买家标识**：下单用户身份。
- **商品清单**：包含商品名称、单价、购买数量。
- **订单总金额**：商品总价。
- **订单状态**：待付款 / 已支付 / 已取消。
- **下单时间**：精确到秒。
- **支付截止时间**：下单时间 + 15分钟。

### 5.2 库存信息
- **商品标识**：SKU 编号。
- **总库存量**：仓库实际货物总数。
- **可售库存量**：当前用户前端可见、可购买的库存数量。
- **预占库存量**：已被未付款订单锁定的商品数量。
- **业务公式**：`可售库存 = 总库存 - 预占库存 - 已售库存`（注：本项目聚焦预占与释放，已售库存非核心关注点）。

### 5.3 支付流水信息
- **流水号**：唯一识别码。
- **关联订单号**：对应哪笔订单。
- **支付金额**：实际支付金额。
- **支付状态**：等待支付 / 支付成功 / 支付失败。
- **完成时间**：收到回调确认的时间。

## 6. 非功能性约束（业务侧）

| 约束类型 | 具体要求 |
| :--- | :--- |
| **时效性** | **订单超时阈值**：严格固定为 **15分钟**。该时长需作为系统配置项以便未来业务调整。 |
| **准确性** | **库存账实相符**：预占库存数与实际未支付订单锁定库存数必须时刻保持逻辑一致，严禁出现负数可售库存。 |
| **幂等性** | 针对同一笔支付回调的重放请求，订单状态仅允许从“待付款”变更为“已支付”一次。 |

## 7. 附录：名词解释

| 术语 | 业务定义 |
| :--- | :--- |
| **预占库存** | 一种临时锁库存机制。用户下单但未付款时，商品并未售出，但为了给该用户留货而暂时冻结的数量。 |
| **支付回调** | 支付渠道处理完用户付款后，主动告知电商平台“这笔钱已到账”的通知动作。 |
| **超卖** | 指系统售出的商品数量超过了实际库存总量，导致无法履约发货的业务事故。本需求通过预占机制规避此风险。 |
