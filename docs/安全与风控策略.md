# 安全与风控策略 (Security & Risk Control Strategy) V1.0

---

## 一、数据一致性策略

### 1.1 原子性操作定义

> **描述**：针对金额、库存等关键数值变更，必须使用数据库原子操作防止并发问题。

#### 库存更新原子性
- **禁止写法**：
  ```sql
  -- 错误：先查询再更新，存在并发风险
  SELECT locked_stock FROM inventory WHERE sku_id = 'SKU001';
  UPDATE inventory SET locked_stock = locked_stock + 10 WHERE sku_id = 'SKU001';
  ```

- **正确写法**：
  ```sql
  -- 正确：使用单条 UPDATE 语句，数据库层面保证原子性
  UPDATE inventory 
  SET locked_stock = locked_stock + 10 
  WHERE sku_id = 'SKU001' 
  AND (total_stock - locked_stock - sold_stock) >= 10;
  ```

#### 库存扣减确认原子性
```sql
-- 正确：一次性完成 locked_stock 减少和 sold_stock 增加
UPDATE inventory 
SET 
    locked_stock = locked_stock - 2,
    sold_stock = sold_stock + 2
WHERE sku_id = 'SKU001'
AND locked_stock >= 2;
```

#### 乐观锁版本控制
```sql
-- 正确：使用 version 字段实现乐观锁
UPDATE inventory 
SET 
    locked_stock = 50,
    version = version + 1,
    updated_at = NOW()
WHERE sku_id = 'SKU001'
AND version = 5; -- 传入的当前版本号
```

### 1.2 事务边界定义

> **描述**：明确哪些业务步骤必须包裹在同一个数据库事务中。

| 业务流程 | 事务包含步骤 | 事务隔离级别 |
|:---------|:-------------|:-------------|
| **订单创建** | 1. 预占库存（库存服务）<br>2. 插入订单表<br>3. 插入订单明细表 | READ_COMMITTED |
| **支付成功** | 1. 更新支付凭证状态<br>2. 更新订单状态<br>3. 库存扣减确认 | READ_COMMITTED |
| **订单取消** | 1. 更新订单状态<br>2. 释放预占库存 | READ_COMMITTED |

**事务异常处理**：
- **回滚规则**：所有运行时异常（RuntimeException）自动回滚
- **提交规则**：业务异常（BusinessException）不回滚（如"库存不足"）
- **超时设置**：事务超时时间 30 秒（Spring `@Transactional(timeout=30)`）

### 1.3 库存账实一致性保障

> **描述**：确保数据库中的库存数据真实反映物理库存。

#### 核心约束
- **可售库存公式**：`可售库存 = 总库存 - 预占库存 - 已售库存`
- **禁止负库存**：任何操作不得导致库存为负数
- **定时对账**：每日凌晨2点执行库存对账任务

#### 对账逻辑伪代码
```
FUNCTION reconcileInventory()
    // 1. 查询所有 SKU 的数据库库存
    dbInventory = db.selectFrom("inventory")
        .where("is_deleted = 0")
        .fetch()
        
    // 2. 计算理论可用库存
    FOR EACH item IN dbInventory:
       理论可用 = item.total_stock - item.locked_stock - item.sold_stock
        
        IF 理论可用 < 0:
            ERROR "库存异常：SKU=" + item.sku_id + 
                  ", total=" + item.total_stock + 
                  ", locked=" + item.locked_stock + 
                  ", sold=" + item.sold_stock
            // 发起人工审核通知
            notifyAuditTeam("库存账实不符：" + item.sku_id)
        END IF
    END FOR
    
    LOG "库存对账完成"
END FUNCTION
```

## 二、接口安全与幂等性

### 2.1 幂等性方案

> **描述**：防止重复请求导致数据异常（如重复扣库存、重复创建订单）。

#### 订单创建幂等（按用户 + SKU + 时间窗口）
- **幂等 Key**：`order:create:{user_id}:{sku_id}:{timestamp_minute}`
- **实现方式**：Redis SETNX + 过期时间（1分钟）
- **Expired**：1 分钟后允许重新下单

#### 库存预占幂等（按订单号）
- **幂等 Key**：`inventory:lock:{order_no}`
- **实现方式**：Redis SETNX + 过期时间（30秒）
- **说明**：订单创建时生成幂等 Key，库存服务收到请求时先检查 Key 是否存在

#### 支付回调幂等（按支付流水号）
- **幂等 Key**：`payment_callback:{payment_no}`
- **实现方式**：Redis SETNX + 过期时间（24小时）
- **说明**：支付渠道回调时，先检查幂等 Key，存在则直接返回成功

**幂等性检查伪代码**：
```
FUNCTION isIdempotent(String幂等Key, Integer ttl, TimeUnit unit) RETURNS Boolean
    // 1. 检查幂等 Key 是否已存在
    IF redis.exists(幂等Key):
        LOG "重复请求（幂等跳过）"
        RETURN false // 表示不应重复处理
        
    ELSE:
        // 2. 设置幂等 Key
        redis.setnx(幂等Key, "1", ttl, unit)
        RETURN true // 表示可以处理
END FUNCTION
```

### 2.2 签名与验签（可选，敏感操作）

> **描述**：防止回调数据被篡改（可选实现）。

#### 签名算法（HMAC-SHA256）
```
// 签名生成
signature = HMAC-SHA256(
    secret_key = "payment_secret_key",
    message = "order_no=ORD20260415001&amount=99.50&status=1"
)

// 接收方验签
received_signature = request.headers["X-Signature"]
calculated_signature = HMAC-SHA256(
    secret_key = "payment_secret_key",
    message = request.body
)

IF received_signature != calculated_signature:
    REJECT request ("签名验证失败")
END IF
```

#### 需要签名的接口
- 支付回调（/api/v1/payment/callback）
- 消费订单状态变更事件

## 三、业务状态机风控

### 3.1 非法流转拦截

> **描述**：在服务层强制拦截非法的状态转移请求。

#### 状态转换校验

```java
// 订单状态机校验（Java 伪代码）
public class OrderStateMachine {
    private static final Map<Integer, Set<Integer>> allowedTransitions = new HashMap<>();
    
    static {
        allowedTransitions.put(0, new HashSet<>(Arrays.asList(1, 2))); // PENDING_PAYMENT -> PAID/CANCELLED
        allowedTransitions.put(1, new HashSet<>()); // PAID -> 终态
        allowedTransitions.put(2, new HashSet<>()); // CANCELLED -> 终态
    }
    
    public static boolean isValidTransition(Integer fromStatus, Integer toStatus) {
        Set<Integer> allowed = allowedTransitions.get(fromStatus);
        if (allowed == null) {
            return false;
        }
        return allowed.contains(toStatus);
    }
}

// 使用示例
IF NOT OrderStateMachine.isValidTransition(order.status, newStatus):
    THROW "非法状态流转：status=" + order.status + " -> " + newStatus
END IF
```

#### 强制校验位置
- **更新订单状态前**：`orderRepository.updateStatus()`
- **支付回调处理时**：`-paymentService.handleCallback()`
- **定时取消订单时**：`orderService.cancelTimeoutOrder()`

### 3.2 状态回滚逻辑

> **描述**：业务失败时，关联资源需回滚到初始状态。

#### 订单创建失败回滚
| 失败步骤 | 回滚动作 |
|:---------|:---------|
| 库存预占成功，但订单创建失败 | 释放预占库存（locked_stock - quantity） |
| 订单创建成功，但事件发布失败 | 记录待重试队列，后续补发 |

#### 支付失败回滚
| 失败步骤 | 回滚动作 |
|:---------|:---------|
| 支付渠道返回失败 | 释放预占库存，更新订单状态为"已取消" |

## 四、异常与边界场景

### 4.1 超时机制

> **描述**：关键业务操作设置 TTL，超时自动触发回滚或通知。

#### 订单超时配置
| 参数类型 | 配置值 | 说明 |
|:---------|:-------|:-----|
| **支付超时** | 15 分钟 | 用户创建订单后未支付的最长等待时间 |
| **库存预占锁** | 30 秒 | Redis 分布式锁的超时时间 |
| **支付回调超时** | 5 秒 | 向支付渠道发起请求的超时时间 |
| **事务超时** | 30 秒 | 数据库事务的最长执行时间 |

#### 订单超时自动处理
```sql
-- 定时任务 SQL（每分钟执行）
SELECT * FROM orders 
WHERE status = 0 -- PENDING_PAYMENT
AND pay_timeout_at < NOW()
AND is_deleted = 0
LIMIT 100;
```

#### 超时后处理流程
1. 更新订单状态为"已取消"（status=2）
2. 发布订单超时事件（MQ）
3. 消费者处理库存回滚
4. 记录超时日志

### 4.2 金额/数据校验

> **描述**：服务端强制校验关键数据，禁止依赖前端传值。

#### 金额校验规则
| 校验项 | 规则 | 错误码 |
|:-------|:-----|:-------|
| **订单金额** | 必须 > 0，且保留 2 位小数 | 1005 |
| **支付金额** | 必须与订单金额完全一致 | 3003 |
| **库存数量** | 必须为正整数，且 <= 最大库存 | 1006 |

#### 金额校验伪代码
```
FUNCTION validateAmount(BigDecimal amount, String field) RETURNS Boolean
    IF amount IS NULL:
        THROW "参数 " + field + " 不能为空", code=1005
    END IF
    
    IF amount.compareTo(BigDecimal.ZERO) <= 0:
        THROW "参数 " + field + " 必须大于0", code=1005
    END IF
    
    IF amount.scale() > 2:
        THROW "参数 " + field + " 保留位数超过2位", code=1005
    END IF
    
    RETURN true
END FUNCTION
```

#### 库存数量校验
```java
// 库存预占前的校验
IF quantity <= 0:
    THROW "购买数量必须大于0", code=1006
    
IF quantity > Integer.MAX_VALUE / 100: // 防止溢出
    THROW "购买数量过大", code=1006
END IF
```

### 4.3 并发冲突处理

> **描述**：针对高并发场景，采用多种策略防止数据异常。

#### 并发预占处理策略
| 策略 | 实现方式 | 场景 |
|:-----|:---------|:-----|
| **Redis 分布式锁** | Lua 脚本 + SETNX | 防止同一 SKU 的并发预占 |
| **数据库乐观锁** | version 字段 + WHERE version = ? | 防止并发更新冲突 |
| **数据库行级锁** | SELECT ... FOR UPDATE | 确保读取后立即更新 |
| **事务隔离级别** | READ_COMMITTED | 防止脏读 |

#### 并发冲突的用户提示
- **锁竞争失败**：提示"网络繁忙，请稍后重试"
- **库存不足**：提示"库存不足，请调整购买数量"
- **订单已存在**：提示"您的订单已创建，请勿重复提交"

### 4.4 数据库异常处理

> **描述**：针对数据库操作的异常情况，制定降级方案。

#### 异常场景及处理
| 异常类型 | 处理方式 | 是否回滚 |
|:---------|:---------|:---------|
| **死锁** | 重试（最多 3 次） | 是（自动） |
| **连接超时** | 重试（最多 2 次） | 是（自动） |
| **主从延迟** | 强制走主库读取 | 否 |
| **事务回滚** | 提示用户重试 | 是（自动） |
| **数据库宕机** | 降级到本地事务日志 | 是（自动） |

#### 本地事务日志（降级方案）
```sql
-- 在本地数据库记录待处理事务
CREATE TABLE local_transaction_log (
    id BIGINT PRIMARY KEY,
    operation_type VARCHAR(32), -- 'LOCK_STOCK' or 'CANCEL_ORDER'
    order_no VARCHAR(32),
    sku_id VARCHAR(64),
    quantity INT,
    status VARCHAR(16), -- 'PENDING' or 'PROCESSED'
    created_at DATETIME,
    retry_count INT DEFAULT 0
);
```

**日志处理流程**：
1. 记录本地事务日志（状态=PENDING）
2. 异步线程尝试恢复事务
3. 成功后更新状态为 PROCESSED
4. 每日清理 7 天前的 PROCESSED 记录

### 4.5 支付回调异常处理

> **描述**：针对支付回调的各种异常场景，制定处理策略。

#### 异常回调处理矩阵

| 异常场景 | 处理动作 | 是否更新订单 | 是否通知人工 |
|:---------|:---------|:-------------|:-------------|
| **金额不符** | 标记为"异常"（status=3） | 否 | 是（必须） |
| **订单已取消** | 拒绝处理，记录日志 | 否 | 否（自动） |
| **重复回调** | 直接返回成功（幂等跳过） | 否 | 否（自动） |
| **回调延迟** | 等待订单超时后取消 | 否 | 否（自动） |
| **签名无效** | 拒绝处理，记录告警 | 否 | 是（必须） |

#### 异常回调处理伪代码
```
FUNCTION handleAbnormalCallback(PaymentCallback callback) RETURNS Response
    payment = findPayment(callback.payment_no)
    order = findOrder(payment.order_id)
    
    IF callback.amount.compareTo(order.total_amount) != 0:
        // 金额不符
        updatePaymentStatus(payment.id, 3) // EXCEPTION
        insertAuditRecord(order.id, "金额不符：订单=" + order.total_amount + 
                          ", 回调=" + callback.amount)
        sendAlertToAuditTeam("支付金额异常：" + order.order_no)
        RETURN error("金额不符", code=3003)
        
    ELSE IF order.status == 2: // CANCELLED
        // 订单已取消
        LOG "订单已取消，支付回调无效: order_no=" + order.order_no
        RETURN success("订单已关闭")
        
    ELSE IF isDuplicateCallback(callback.payment_no):
        // 重复回调
        LOG "重复回调（幂等跳过）: payment_no=" + payment.payment_no
        RETURN success("重复回调")
        
    END IF
    
    // 其他异常场景...
END FUNCTION
```

### 4.6 日志与监控

> **描述**：关键业务操作必须记录日志，并设置告警阈值。

#### 关键日志清单
| 日志类型 | 关键字段 | 级别 |
|:---------|:---------|:-----|
| **订单创建** | order_no, user_id, sku_id, amount | INFO |
| **库存预占** | sku_id, quantity, lock_id | DEBUG |
| **支付回调** | payment_no, order_no, amount, status | INFO |
| **超时取消** | order_no, pay_timeout_at | WARN |
| **并发冲突** | sku_id, error_type | ERROR |
| **金额不符** | order_no, expect_amount, actual_amount | ERROR |

#### 告警规则
| 告警规则 | 告警级别 | 处理方式 |
|:---------|:---------|:---------|
| **并发冲突率 > 5%** | WARN | 优化锁粒度或增加重试 |
| **库存为负数** | CRITICAL | 立即人工介入 |
| **支付金额不符** | ERROR | 人工审核 |
| **数据库连接池耗尽** | CRITICAL | 通知运维扩容 |
